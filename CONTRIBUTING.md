# Contributing to protowire-java

Welcome — this is the Java port of [protowire](https://protowire.org), a
language-neutral wire-format toolkit. It tracks the canonical specification
in [`trendvidia/protowire`](https://github.com/trendvidia/protowire) and is
one of nine sibling ports (Go, C++, Rust, Java, TypeScript, Python, C#,
Swift, Dart).

> **Steward integration is rolling out.** The governance described in
> [GOVERNANCE.md](GOVERNANCE.md) (and the upstream
> [`governance.pxf`](https://github.com/trendvidia/protowire/blob/main/governance.pxf))
> is the steady-state model. While Steward is being finalised, pull
> requests are reviewed by human maintainers in the conventional
> way — open a PR, expect review, iterate.

## Where bugs go

| Symptom | File against |
|---|---|
| Java-only crash, wrong API ergonomics, performance regression in this port only | `trendvidia/protowire-java` |
| The same input produces different output here vs another port | upstream [`trendvidia/protowire`](https://github.com/trendvidia/protowire) (cross-port wire-equivalence regression) |
| Spec / grammar / proto annotation question | upstream [`trendvidia/protowire`](https://github.com/trendvidia/protowire) |
| Decoder crash / hang / OOM on adversarial input | **email security@trendvidia.com**, do not file public issue (see [SECURITY.md](SECURITY.md)) |

## Local development

Java 21 toolchain is required. Gradle downloads and pins it automatically.

```sh
./gradlew build              # compile + test all 8 modules
./gradlew :pxf:test          # only the PXF module
./gradlew :pxf:jar           # produce the runtime jar (used by the
                             # JetBrains plugin in the spec repo)
./gradlew jacocoTestReport   # coverage report under <module>/build/reports/jacoco/
```

The protobuf classes are generated by `protobuf-gradle-plugin` against
`protoc 3.25.5` and the `proto-annotations` from upstream. You don't need
`protoc` on PATH; Gradle fetches the right one.

## Sending changes

1. Open a draft PR early — it costs nothing and unblocks design feedback
   before you finish.
2. **For changes that touch the parser/encoder behaviour**: after the
   draft is up, please post a comment listing which port-specific test
   cases you exercised. The cross-port wire-equivalence promise means a
   wrong move here can break six other ports' contracts.
3. **For changes that touch the wire format itself** — annotation field
   numbers in `proto/`, the PXF grammar, the SBE schema-id semantics —
   open the upstream PR in [`trendvidia/protowire`](https://github.com/trendvidia/protowire)
   first. This port shouldn't lead spec changes; it implements them.

## Code style

- Java 21 features are fair game (records, sealed types, pattern matching
  in switch, virtual threads where it actually helps).
- Public API: javadoc every public type and method; if it's `public` it's
  part of the SemVer contract.
- Keep allocations on the hot path low; the SBE module has a
  zero-allocation `View` API for a reason.
- Match `-Xlint:all` cleanliness; the build flags it.

## What we don't accept

- Changes that break wire-equivalence with another sibling port.
- Changes that ship `protoc` plugins of their own (codegen lives in the
  spec repo's `cmd/protoc-gen-pxf-java-meta/`, not here).
- New top-level dependencies without a one-line justification in the PR.

## Releases

This port releases in lockstep with the rest of the `protowire-*` stack.
The version line is `0.70.x` for the first coordinated public release;
ports that share a `0.70.x` minor implement the same wire contract.

A maintainer cuts releases by:
1. Bumping `version` in `build.gradle.kts`.
2. Adding a `## [X.Y.Z]` section to `CHANGELOG.md`.
3. Tagging `vX.Y.Z` on `main`.
4. The `.github/workflows/publish.yml` workflow promotes the artifacts
   to Maven Central (under the `org.protowire` namespace).
